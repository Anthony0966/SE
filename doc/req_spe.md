# [미니 드라이브] 요구사항 정의서

## 1. 서론

### 1.1 이 문서 목적
이 문서는 사내 협업 환경 개선을 위한 구축형 클라우드 스토리지 시스템 '미니 드라이브(Mini Drive)' 프로젝트의 개발을 위해 기능적, 비기능적, 인터페이스 요구사항을 명확히 정의하는 문서이다. 본 정의서는 설계 및 구현 단계의 기준이 되며, 품질 목표 달성을 위한 검증 지표로 활용된다.

### 1.2 프로젝트 개요

#### 1.2.1 프로젝트 정의
분산된 파일 공유 방식으로 인한 버전 관리 혼선과 데이터 유출 리스크를 방지하기 위해, 200명 규모의 사내 직원이 동시 접속 가능한 웹 기반의 On-premise식 클라우드 저장소 시스템을 구축한다.

#### 1.2.2 주요 기능
| ID | 설명 |
| :--- | :--- |
| F01 | 대용량 설계 파일의 안정적인 업로드/다운로드 (Chunk Upload 지원) |
| F02 | 트리 구조의 폴더 관리 및 휴지통 복구 기능 |
| F03 | 메타데이터 기반의 빠른 파일 검색 및 필터링 |
| F04 | 파일 수정 시 이전 버전 이력 보관 및 롤백 기능 |
| F05 | 사용자 간 접근 제어 및 기간 만료형 외부 공유 링크 생성 |

### 1.3 용어 정의
| 용어 | 설명 |
| :--- | :--- |
| 메타데이터 | 데이터에 관한 데이터로, 본 시스템에서는 파일명, 업로드 날짜, 크기, 업로더 등의 정보를 의미함 |
| Chunk Upload | 대용량 파일을 작은 단위(Chunk)로 쪼개어 전송함으로써 네트워크 불안정 시에도 재전송 효율을 높이는 방식 |
| On-premise | 소프트웨어 등 솔루션을 기업의 자체 인프라 장비에 직접 설치하여 운영하는 방식 |
| SPA | Single Page Application의 약자로, 단일 페이지에서 브라우저 렌더링을 처리하여 사용자 경험을 향상시키는 웹 애플리케이션 구조 |

## 2. 요구사항

### 2.1 기능적 요구사항

#### 2.1.1 사용자 및 권한관리 기능
| 분류 | 요구사항 | 우선순위 |
| :--- | :--- | :--- |
| FR-001 | 사용자는 시스템 접근 시 아이디와 비밀번호를 통해 Auth를 거쳐야 한다. | 상 |
| FR-002 | 관리자는 사용자 그룹을 생성하고 사용자별 스토리지 할당량을 관리할 수 있다. | 중 |

#### 2.1.2 파일 및 폴더 관리 기능
| 분류 | 요구사항 | 우선순위 |
| :--- | :--- | :--- |
| FR-003 | 사용자는 웹 브라우저를 통해 문서, 이미지, 대용량 파일을 업로드 및 다운로드할 수 있다. | 상 |
| FR-004 | 시스템은 대용량 파일 업로드 시 Chunk Upload 방식으로 파일을 분할 전송해야 한다. | 상 |
| FR-005 | 사용자는 다중 레이어의 트리 구조로 폴더를 생성하고 이름을 변경할 수 있다. | 상 |

#### 2.1.3 검색 및 공유 기능
| 분류 | 요구사항 | 우선순위 |
| :--- | :--- | :--- |
| FR-006 | 사용자는 메타데이터(파일명, 업로드 날짜, 업로더, 유형)를 기반으로 파일을 필터링 검색할 수 있다. | 상 |
| FR-007 | 사용자는 내부 사용자에게 파일 조회, 수정, 댓글 작성 권한을 부여할수 있다. | 상 |
| FR-008 | 사용자는 외부 공유 시 보안을 위해 만료 기간이 설정된 다운로드 암호화 링크를 생성할 수 있다. | 상 |

#### 2.1.4 버전 및 복구 기능
| 분류 | 요구사항 | 우선순위 |
| :--- | :--- | :--- |
| FR-009 | 시스템은 파일 수정 시 Differential Storage 방식을 고려하여 이전 버전 이력을 보관한다. | 상 |
| FR-010 | 사용자는 특정 시점의 파일 버전으로 롤백 하거나 다운로드할 수 있다. | 상 |
| FR-011 | 시스템은 파일 삭제 시 즉시 영구 삭제하지 않고 휴지통으로 이동시켜 복구 기능을 제공한다. | 중 |

### 2.2 비기능적 요구사항

#### 2.2.1 운영 환경에 대한 요구사항
| 분류 | 요구사항 | 우선순위 |
| :--- | :--- | :--- |
| NFR-001 | 시스템은 별도의 클라이언트 설치 없이 크롬 기반 웹 브라우저에서 동작해야 한다. | 상 |
| NFR-002 | 시스템은 사내망 환경(On-premise) 내부 서버에 구축되어야 한다. | 상 |

#### 2.2.2 성능 요구사항
| 분류 | 요구사항 | 우선순위 |
| :--- | :--- | :--- |
| NFR-003 | 시스템은 200명 동시 접속 환경에서 안정적인 쓰루풋을 보장해야 한다. | 상 |
| NFR-004 | 1GB 크기의 단일 파일 업로드는 사내 1Gbps 망 기준 30초 이내에 완료되어야 한다. | 상 |
| NFR-005 | 메타데이터 기반 파일 검색 시 API 응답 시간은 0.5초 이내여야 한다. | 중 |

#### 2.2.3 신뢰성 및 보안 요구사항
| 분류 | 요구사항 | 우선순위 |
| :--- | :--- | :--- |
| NFR-006 | 데이터 전송 구간은 패킷 도청 방지를 위해 HTTPS 프로토콜을 필수적으로 적용한다. | 상 |
| NFR-007 | 데이터베이스 내의 사용자 계정 정보는 암호화하여 저장한다. | 상 |
| NFR-008 | 파일 전송 중 네트워크 단절 감지 시 최대 3회 Auto retry를 수행한다. | 상 |
| NFR-009 | 청크 병합 시 SHA-256 알고리즘을 통한 무결성 검증을 100% 통과해야 한다. | 상 |
